Problem

Sumo Logic comes into the picture when you are dealing with a lot of data, say terabytes or even petabytes. However, there was a core issue as users were lost trying to figure out how to write metrics queries.







As a Product Designer at Sumo Logic, my work makes Andre's and Kathy's lives simpler. Working closely with UX Research, Product Management and Engineering, I scoped out the specifics of the problem. To summarize,


1 Sumo is extremely powerful only if you know how to use it.

2 Users are lost trying to figure out how to write powerful queries that help them maximize use.

3 Sumo handles both structured and unstructured data that complicates the semantics of the query language.

4 Query language needs to be simplified.

5 Users write generic queries that fetch all results as they cannot understand how to write queries.

6 Sumo processes a lot of data (in petabytes).

7 Users tend to write multiple queries at once to get insights about their complex problems.

Design Specifics


Based on qualitative data from our user research sessions, we were able to identify 4 major emerging themes:

Simplifying Complexity

Both Kathy and Andre found it hard to write metric queries during critical times. In order to simplify the query language, I started looking into the semantics of the platform's query language. Talking to our internal users gave us insights into what really mattered in the query language and how people grouped data. While some users preferred simpler lanugage, others were boggled by what those categories even meant. We observed that even the order of these mattered as they resulted in different kind of queries and had serious backend implications. By the end of it, we broke it down into four broad categories: Metric, Filter, Aggregation, Group By.

Initial State with four categories


Sense of Freedom

When we spoke to our users who were like Kathy or Andre, we found that neither knew what to start typing. It was clear that we needed to have an autocomplete component that handles their unique cases. Through informal interviews and query writing observations, we found that users wanted the freedom to write any kind of query and still have some structure. Deciding the level of freedom that users should have when using a structured query builder was the core question.



Autocomplete Behavior Conceptual Designs


Interactions

With the introduction of the structured query builder, there were key concerns that had to be addressed - "How do we switch to the other one? Does this tool handle all kinds of queries? I want to be able to modify queries using the query language too!"

While this was an essential function, the users neither wanted it to stand out explicitly or be hidden within menus. This led to finer interaction design questions.

Harsh Reality

We found that Kathy and Andre leveraged multiple complex queries to solve their respective problems. Error handling and accessibility were other factors that we looked into at this point.

Outcome



With the help of the new solution, Kathy was able to successfully figure out new routes that had to be introduced based on trending tourism spots, social media tags and her airline's passenger traffic during various times of the year. In fact, she was able to create dashboards that gave her rich information at an international scale.

On the other hand, Andre was able to single-handedly pinpoint what led to the system breakdown and get the systems up and running without having to alert another soul.

This new query experience led to happy users, increased satisfaction levels, higher retention rates and better task completion success rates by 20%.
